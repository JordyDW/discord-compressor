# Discord Compressor

<p align="center">
  <img src="assets/icon-src/icon-rounded.png" width="128" alt="Discord Compressor app icon" />
</p>

An Android (Expo / React Native) app that shrinks a phone video to fit Discord's
**free 10 MB upload limit** — no Discord Nitro, no server Boosts.

Discord lowered the free per-file cap from 25 MB to **10 MB** in late 2024, so most
phone clips (1080p/4K, 30–60 fps) are too big to post in a DM or unboosted server.
This app fixes that on-device.

## Two ways to use it

1. **Share into it (the "automatic" flow):** in your gallery/camera, tap
   **Share → Discord Compressor** — one clip or several at once (multi-select). It
   compresses each, then re-opens the Share sheet so you pick Discord. _(No app can
   hook Discord's own upload button — Discord is closed — so this Share-sheet hop is
   the realistic "automatic" path.)_
2. **Standalone:** open the app, **Pick videos** (one or many), compress, send.

**Batch:** picking/sharing multiple videos compresses them one after another and
shows a per-clip results list with a **Send all to Discord** button (one Share sheet,
all files). You can **remove** a clip from the queue while it's still waiting, or drop
one from the results before sending. Either way it also **saves a compressed copy of
each to your gallery**.

## How the compression works

On-device, FFmpeg-free (FFmpegKit was retired in 2025). The engine is
[`react-native-compressor`](https://github.com/numandev1/react-native-compressor)
(native MediaCodec, H.264/AAC MP4 out — what Discord plays most reliably).

Hitting an exact size cap from a single bitrate guess is unreliable, so
[`src/lib/compress.ts`](src/lib/compress.ts) runs an **estimate → transcode →
verify → retry** loop:

- [`src/lib/encodePlan.ts`](src/lib/encodePlan.ts) — pure, unit-tested math:
  spread a 10 MB budget (×0.92 safety) across the clip's duration to get a target
  bitrate, reserve headroom for audio, and step resolution down a ladder
  (1280 → 854 → 640) as the budget tightens.
- After each pass it re-reads the real output size; if still over, it backs the
  bitrate off and retries (up to 3 passes).
- A clip too long to fit even at the floor bitrate (350 kbps) is reported as
  **"too long — trim it"** instead of silently producing an over-limit file.

## Tech stack

Runs on **Expo SDK 57** (React Native 0.86, React 19.2, new architecture).

| Concern | Package |
| --- | --- |
| Compression | `react-native-compressor` **2.x** (Nitro) |
| Receive shares | `expo-share-intent` (config plugin — see note) |
| Re-share out | `react-native-share` |
| Save copy | `@react-native-camera-roll/camera-roll` |
| Pick video | `expo-image-picker` |
| Keep screen awake | `expo-keep-awake` |

> **Dependency note — the SDK 57 share-target swap:**
> SDK 57 (RN 0.86) needs `react-native-nitro-modules` 0.36, but
> `react-native-nitro-share-intent@0.5` (the old share receiver) is pinned to Nitro
> `^0.31` and unmaintained — it can't compile against RN 0.86. So the share target
> now uses **`expo-share-intent`**, a config-plugin library with **no Nitro
> dependency**. That frees the compressor to move to **2.x** (Nitro 0.36), which is
> the line built for current React Native.
>
> `expo-share-intent@7`'s peer range still says `expo ^56`, so:
> - install with **`--legacy-peer-deps`** (pinned in [`.npmrc`](.npmrc)), and
> - a `package.json` **`overrides`** block forces its `expo-constants` /
>   `expo-linking` to the SDK 57 copies (avoids duplicate native modules).
>
> Both are drop-in: the compressor's JS API (`Video.compress`, `getVideoMetaData`,
> `getRealPath`) is unchanged in 2.x, and `expo-share-intent`'s `useShareIntent()`
> hook covers both cold- and warm-start shares.

The Android **share target** is generated by the `expo-share-intent` config plugin
in [`app.json`](app.json) (`SEND` / `SEND_MULTIPLE` for `video/*`) during prebuild.

## Build & run

Requires Node + the Android SDK (`adb`, build-tools) and a connected device/emulator.

### Why Expo Go can't run this

Expo Go is a fixed, pre-built app: it only contains the Expo SDK modules + pure JS.
This app's core is **native modules that aren't in it** — the compressor
(`react-native-compressor` 2.x is a **Nitro** module), the share-target receiver,
`react-native-share`, and camera-roll. Opened in Expo Go it crashes immediately with
**`Failed to get NitroModules`**, and there's no fix — video transcoding needs a
native codec Expo Go doesn't ship. You need a build of *this* app (below). A
**development build** gives the identical Expo-Go workflow (`expo start`, QR, hot
reload) with the native modules baked in — install it once, then iterate over-the-air.

### Install dependencies

```bash
npm install     # .npmrc pins legacy-peer-deps=true (see the dependency note above)
npx expo prebuild -p android      # generates ./android from app.json (icons, manifest, plugins)
```

### Development build — hot-reload dev loop (the "Expo Go equivalent")

Build + install the dev client once, then use Metro over USB:

```bash
cd android && ./gradlew :app:assembleDebug -PreactNativeArchitectures=arm64-v8a && cd ..
adb install -r android/app/build/outputs/apk/debug/app-debug.apk
npm run start     # = adb reverse tcp:8081 + expo start --dev-client --localhost
```

Open **Discord Compressor** on the phone → it connects to Metro → edit JS/TSX, save,
hot reload. Rebuild/reinstall only when **native** deps change.
_(Shortcut: `npx expo run:android` does build + install + Metro in one step.)_

### Standalone release APK — offline daily use (no laptop/Metro)

Self-contained (Hermes bytecode embedded, debug-signed), runs with no dev server:

```bash
cd android && ./gradlew :app:assembleRelease -PreactNativeArchitectures=arm64-v8a && cd ..
adb install -r android/app/build/outputs/apk/release/app-release.apk
```

Notes:
- `arm64-v8a` targets modern phones and keeps the APK small; drop the `-P` flag to
  build all ABIs (`armeabi-v7a, arm64-v8a, x86, x86_64`).
- Debug and release share the app id `com.jordydw.discordcompressor` and the debug
  signing key, so **installing one replaces the other** on the device.
- For a cloud build instead: `eas build --profile preview -p android` (see [`eas.json`](eas.json)).

### iOS — EAS Build (macOS or cloud)

iOS builds require Xcode (macOS only) or [EAS Build](https://docs.expo.dev/build/introduction/)
(Expo's cloud builder). You also need an [Apple Developer account](https://developer.apple.com)
for signing.

**Cloud build (recommended — works from any OS):**

```bash
eas build --profile preview -p ios          # ad-hoc IPA for physical device
eas build --profile preview-simulator -p ios  # simulator build for local testing
eas build --profile production -p ios       # App Store build
```

**Local build (macOS with Xcode):**

```bash
npx expo prebuild -p ios   # generates ./ios from app.json
npx expo run:ios           # build + launch in simulator
```

The iOS share extension (appears in the system Share sheet when you share a video from
Photos or Files) is configured via `expo-share-intent`'s `iosActivationRules` in
[`app.json`](app.json). It accepts up to 5 videos at once and uses the app group
`group.com.jordydw.discordcompressor` to hand URIs to the main app.

## Manual end-to-end check (the real success criteria)

1. Film a **>10 MB** clip → Share → **Discord Compressor** → output is **<10 MB**,
   a copy lands in the gallery, and it **uploads + plays in Discord**.
2. Standalone: open app → pick the same clip → same result.
3. A long 4K clip → shows the graceful **"too long, trim it"** message.

## Roadmap

### Shipped

- [x] On-device size-targeting compression (**< 10 MB**) — estimate → transcode →
      verify → retry loop with a resolution ladder
- [x] Share target (`SEND` / `SEND_MULTIPLE`) + standalone picker
- [x] **Batch / multi-video** with a removable queue and **Send all to Discord**
- [x] Saves a compressed copy of each clip to the gallery
- [x] Custom app icon, safe-area layout, dark Discord-flavoured UI
- [x] Expo SDK 57 / React Native 0.86

### Planned

- [x] **Target-size presets** — 10 / 25 / 50 MB chips on the home screen; per-clip
      override in the batch queue
- [x] **Adaptive audio budget** — audio drops from 128 → 96 → 64 kbps as the total
      bitrate budget shrinks, giving more bits to video on long clips
- [x] **Cancel in-progress encode** — Cancel button aborts the running job and resets
      to idle, discarding remaining queued clips
- [x] **Before/after thumbnail preview** — thumbnails generated after each encode and
      shown side-by-side on the result screen and in batch rows
- [x] **Per-clip target overrides** — 10 / 25 / 50 MB chips on each queued clip in
      WorkingScreen's "Up next" list
- [x] **Trim-to-fit** — when a clip is too long for any quality, a "Trim to fit (Xs) →"
      button silently cuts the video to the maximum fitting duration and re-compresses

### Longer term

- [x] **iOS** support — `bundleIdentifier`, `NSPhotoLibrary` permissions, share-extension
      `iosActivationRules` for `video/*`, and EAS Build profiles all configured; requires
      an Apple Developer account + macOS/EAS to build
- [ ] Universal / release-signed APK builds for wider distribution
- [ ] Optional backend transcode fallback for clips too big for on-device
